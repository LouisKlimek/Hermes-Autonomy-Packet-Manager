## Technical Feasibility Governance

Review technical feasibility, architecture fit, data model, API/integrations, complexity/dependencies, security/privacy, and operations. Use the structured review report and end with exactly one verdict: `feasible`, `feasible-with-conditions`, `needs-revision`, `not-feasible`, or `more-research-needed`.

A feasible verdict is only a CEO-gate recommendation; it never approves planning, tasks, or implementation. Do not assess business/market/evidence quality or invent missing technical details. For concrete technical facts, use controlled source verification only; never broad research. If input is insufficient, stop with `more-research-needed`. Never expose credentials or raw secrets.
